2. 先用人话解释：网站到底是怎么工作的

你可以把一个网站想成“前台 + 后台 + 仓库 + 外部服务”。

2.1 前台是什么

前台就是用户看到的页面，比如：

首页
班级仪表盘
个人中心
每日更新页
资源共享页

前台负责两件事：

展示东西给用户看
把用户点击、填写、上传这些动作发出去

但前台不应该直接保存核心数据，也不应该直接保存 API Key。

2.2 后台是什么

后台就是网站后面的服务器。

它负责：

验证“这个人是谁”
判断“他有没有权限做这件事”
读数据库
写数据库
调 AI
返回结果给前台

所以你可以理解成：

前台是窗口，后台是大脑。

2.3 数据库是什么

数据库就是仓库。

它负责存：

用户账号
班级信息
学生每日更新
测试记录
资源内容
5 分率结果
AI 配置
操作日志
2.4 外部服务是什么

外部服务就是你接入的大模型，比如别人的 AI 平台。

比如你后台里配置了一个 Provider，那后台就会拿着：

Base URL
API Protocol
API Key
Model ID

去请求那个外部 AI 服务，然后把结果拿回来。

3. 为什么 API Key 一定不能放前台

这个点非常重要。⚠️

3.1 原因很简单

如果你把 API Key 放在前台，等于任何用户都能看到。

因为前台代码会发送到用户浏览器里，用户是可以打开开发者工具看到请求内容的。

3.2 一旦放前台会发生什么
用户可能复制你的 Key
别人可能盗刷你的额度
你可能被恶意调用
费用可能暴涨
安全彻底失控
3.3 正确做法

必须是：

管理员在后台配置 API
API Key 只存后端
前台用户永远看不到真实 Key
前台只看到 AI 功能结果，不看到配置细节

这就是大部分网站的正常做法。

4. 你最关心的：前台用户怎么“进后台”

这里最容易绕，我用最简单方式讲。

4.1 不是“前台登录到后台”

更准确地说，是：

用户先登录网站，然后前台带着登录凭证，向后台发请求。

后台收到后，会判断：

你是谁
你是什么角色
你能不能访问这个接口
4.2 一个完整例子

比如学生 Billy 登录后，进入“每日更新”页面，提交今天的学习情况。

流程是：

Billy 在前台输入账号密码登录
后台验证成功后，发一个登录凭证给前台
这个凭证通常叫 token / session
前台把这个凭证保存起来
Billy 点击“提交更新”
前台把“今天做了什么”发给后台，同时带上凭证
后台看到凭证，知道这是 Billy
后台把数据写进数据库
后台再调用 AI 和评分规则
后台把新的 5 分率结果返回给前台
前台把结果显示出来

所以不是“前台跑到后台”，而是：

前台发请求给后台，后台处理后再回给前台。

5. 你可以把整个系统理解成这条链路
用户动作链路

用户打开网页
→ 前台页面展示
→ 用户点击 / 填表
→ 前台发请求到后台
→ 后台验证权限
→ 后台查数据库
→ 后台必要时调用 AI
→ 后台计算结果
→ 后台返回数据
→ 前台更新页面

这就是一个网站最基本的运行方式。

7. 为什么一定要做“AI 适配层”

你说的这种配置：

Provider
Base URL
API Protocol
API Key
Model ID
Model Name

本质上说明你不是想只绑定某一家 AI。
你是想做一个可切换模型供应商的系统。

这时就必须做一个“AI 适配层”。

7.1 AI 适配层是干嘛的

它负责把“不同家的接口差异”统一起来。

比如不同 Provider：

请求地址不一样
请求体格式不一样
返回格式不一样
鉴权方式可能有差别
有的走 OpenAI 风格协议，有的不是

AI 适配层的作用就是：

前面的业务逻辑永远用同一种调用方式，底层再去适配不同 Provider。

7.2 不做适配层会怎样

如果不做适配层，以后每换一个模型，你都得改很多业务代码。
那会非常乱，也很难维护。

11.2 正确调用方式示例
错误方式

清洗返回结果
返回前台可显示内容

这样用户永远看不到真实密钥。

15. 5 分率模块如何接入 AI

你前面说的逻辑本质上是：

有测试成绩时，以历史成绩 + 趋势 + 稳定性为主
没测试但有复习描述时，AI 判断复习质量
什么都没有时，做衰减
高水平稳定者，模糊输入影响较小
输入很模糊，比如只写“复习了”，只能微调甚至不抵消衰减

所以建议这样分工：

15.1 规则引擎负责

趋势计算
稳定性计算
每日衰减
上限和下限控制
15.2 AI 负责
解析学生文字描述
判断复习深度
生成解释
生成建议
产出一个“学习质量分”
15.3 最终结果

最终 5 分率 = 规则结果 + AI 小幅修正

不要让 AI 直接裸算整套概率。
这样最稳。

16. 登录和权限执行手册

你刚刚特别问“网站是怎么登录到后端的”，这里给你写成可执行流程。

16.1 登录流程
用户打开登录页
输入账号密码
前台调用 /api/auth/login
后台验证成功
后台返回 token 或 session
前台保存登录状态
以后每次请求都带上凭证
后台按凭证判断用户身份和角色
16.2 权限判断流程

以管理员打开 AI 设置页为例：

管理员访问 /admin/ai
前台请求 /api/auth/me
后台返回角色：admin
前台允许进入后台页面
请求 /api/admin/ai/providers
后台再次校验角色
校验通过才返回数据

这里要注意：
前台拦截只是体验层，真正安全要靠后端校验。

也就是说，哪怕有人强行访问管理员接口，只要他不是 admin，后端也必须拒绝。

13. 关键流程说明
13.1 管理员新增一个 AI Provider 的流程
管理员登录后台
打开“AI 设置”
进入“新增 Provider”
填写模板对应字段
保存
系统加密 API Key 并写入数据库
管理员点击“测试连接”
测试通过后可启用
在“模型用途绑定”中把该模型绑定到某个业务场景
13.2 学生提交每日更新时的 AI 调用流程
学生提交每日更新
前端调用平台后端接口
后端写入 daily_updates
后端根据 scene_code = daily_update_analysis 查找当前绑定模型
后端读取对应 Provider 配置
后端调用外部模型
得到 AI 质量评分、解释和建议
后端再结合规则引擎更新 5 分率
结果写入 probability_snapshots
前端刷新显示
14. 验收标准

下面这部分很适合你后续盯开发。

14.1 功能验收
管理员可新增 Provider
管理员可编辑 Provider
管理员可测试连接
管理员可启用/停用 Provider
管理员可配置业务场景绑定模型
学生端无法看到任何 API Key
非管理员无法访问 AI 配置接口
AI 调用日志可追踪
14.2 安全验收
数据库中 API Key 为加密存储
前台接口不返回真实 API Key
所有 /api/admin/* 接口均有角色校验
配置改动有审计日志
14.3 系统验收
前端不直接请求第三方模型地址
模型切换后无需改前端代码
场景绑定生效后，调用走新的 Provider
测试连接结果可正确反馈成功/失败原因

16. 给你一句最关键的落地结论

这个系统最正确的做法是：

让管理员在后台配置 AI，而不是让前台用户配置；让前端只访问你自己的后端，而不是直接访问外部模型；让后端通过统一的 Provider 配置和适配层去调用不同 AI 服务。

这样你这个产品后面才会：

安全
好维护
好切换模型
好控权限
好查问题
