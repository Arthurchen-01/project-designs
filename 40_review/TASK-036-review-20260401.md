## 2. 加密实现 ⚠️ 有安全隐患

### 正确之处 ✅
- 使用 AES-256-GCM（认证加密）— 业界标准
- IV 长度 12 字节 — GCM 推荐值
- 每次加密随机 IV — 避免 nonce 重用
- 密文格式 `iv:ciphertext:authTag` — 清晰可解析
- `maskApiKey` 解密失败时返回 `****` — 不泄露信息

### 安全问题 🔴

#### M1: 密钥派生使用裸 SHA-256，非 KDF

```ts
return createHash('sha256').update(secret).digest()
```

**问题：** SHA-256 是快速哈希，不适合作为密钥派生函数。如果 `AI_KEY_ENCRYPTION_SECRET` 熵值不够高（如用户设置的短密码），容易被暴力破解。

**建议：** 改用 `scrypt`（Node.js 内置）或 PBKDF2 with ≥100,000 iterations：
```ts
import { scryptSync } from 'crypto'
return scryptSync(secret, 'ap-tracker-salt', 32)
```

#### M2: 无密钥轮换机制

如果 `AI_KEY_ENCRYPTION_SECRET` 泄露或需要更换，现有加密数据无法批量重新加密。建议后续支持密钥轮换。

#### L3: maskApiKey 泄露密钥长度信息

`sk-****xyz` 格式暴露了前 4 + 后 3 字符，对长密钥风险低，但对短密钥可能有用。

## 3. ai-config.ts ✅ fallback 策略合理

### 三级 fallback 逻辑

```
场景路由 → 默认 DB Provider → 环境变量
```

- `getAIConfigForScene(sceneCode)` — 按场景查找 → fallback 默认 → fallback env ✅
- `getAIConfigAsync()` — 默认 DB → env ✅
- `getAIConfig()` — 同步版，仅 env（向后兼容）✅

**`getConfigFromDB` 捕获所有异常后返回 null** — 保证 DB 不可用时不影响系统运行，合理。

### 潜在问题 ⚠️

- `decryptApiKey` 在 DB 读取时被调用，如果加密密钥变了（M2 问题），会静默失败并 fallback 到 env，可能让用户困惑。
- 无缓存机制：每次调用都查 DB + 解密。对高频调用场景可能有性能影响。建议加 TTL 缓存。

### providers/[id]/route.ts
- GET/PUT/DELETE 均有存在性检查 ✅
- PUT 中 `!body.apiKey.includes('****')` 判断是否需要重加密 — 聪明的做法 ✅
- **无鉴权检查 🔴**

### routing/route.ts
- PUT 使用 for 循环逐个 upsert — 对 SQLite 可接受（不支持交互式事务）
- **无鉴权检查 🔴**

### 🔴 关键安全问题：Admin API 完全无鉴权

所有 `/api/admin/ai/*` 路由均未检查：
- 用户是否已登录
- 用户是否为 admin 角色
- CSRF 保护

**这是当前最大的安全隐患。** 任何知道 URL 的人都可以：
1. 查看所有 AI Provider 配置（掩码后的 key）
2. 添加/修改/删除 Provider（包括注入恶意 API endpoint）
3. 修改路由规则，将 AI 调用导向任意服务器

**建议立即添加中间件：**
```ts
// 至少检查 cookie 中的用户角色
const user = await getUserFromCookie(req)
if (!user || user.role !== 'admin') {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 403 })
}
```
