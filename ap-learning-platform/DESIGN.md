# AP 多模态 AI 学习伴侣 — 完整设计方案

> 基于学习目标 + 网站架构师需求 + could-coding 现有项目
> 设计日期：2026-03-29

---

## 一、产品定位

这个网站不只是一个 AP 刷题网站，而是要进化成一个：

**以 Bluebook 模考体验为外壳，以多模态 AI 学伴为核心，以"提问—理解—追问—记录—复习"为主线的 AP 陪伴式学习系统。**

核心价值观：
- 鼓励理解、追问、反思，而不是只鼓励刷题
- 对概念的真实理解
- 对知识漏洞的持续暴露
- 对问题的主动提出
- 对底层逻辑的苏格拉底式掌握
- 对学习过程的可记录、可回放、可评分

---

## 二、产品形态

### 形态 1：站内沉浸式学习台

- 左侧看 PDF / 讲义 / 题目
- 右侧 AI 老师陪你学
- 可拖拽布局
- 可语音输入
- 可结合页面内容发问

### 形态 2：全局悬浮插件

- 在任意学习场景里调出
- 半透明、轻量、不挡视线
- 可随时输入、语音、提问、对话
- 配置自己的 provider / key / base url

## 三、核心功能模块

### 模块 1：套题模考系统

**现状**：已有 Bluebook 风格界面，支持 MCQ/FRQ/计时器/题号导航

**升级方向**：
- 接入更多题库数据（8 科目 × 多年份）
- 支持 Part A/B 切换
- 增加 Review 页面
- 增加 Scratchpad 草稿板
- 做题进度云端同步

---

### 模块 2：PDF 伴读工作台（核心创新）

**目标**：真正的桌面级学习工作台

**功能要求**：
1. 用户上传 PDF 后，左侧显示真正的 PDF 阅读区
2. PDF 阅读区支持上下滚动浏览多页内容
3. 页面整体采用左右分栏布局：
   - 左边：PDF 阅读区
   - 右边：AI 老师对话区
4. 左右区域之间支持拖拽调整宽度
5. 不强制 1:1，用户可以自由改变比例
6. 拖拽后的布局状态本地记忆

**技术方案**：
- PDF 渲染：使用 pdf.js（Mozilla 开源库）
- 分栏布局：CSS Flexbox + 自定义拖拽手柄
- 状态管理：localStorage 保存布局比例

---

### 模块 3：静默识别学习流（核心创新）

**目标**：先识别，不说话，等用户发问时再带上下文回答

**工作流程**：
1. 用户打开 PDF / 讲义页面
2. 用户点击"预读当前页"按钮
3. 系统把当前页面截图送给视觉模型
4. AI 先只做识别和结构化理解
5. 识别结果缓存在本地状态中
6. 这一步不立刻输出回答
7. 用户之后提问时，系统自动把用户问题 + 页面识别结果一起送入模型
8. AI 基于这份"预读上下文"回答

**适用场景**：
- 物理题图很复杂，语言难描述
- 数学公式多，纯文本难表达
- 新学内容时先看讲义，再发问

---

### 模块 4：全局悬浮插件（核心创新）

**目标**：AI 从"页面"变成"系统级助手"

**功能要求**：
1. AI 以浮动方框形式存在
2. 可以拖拽位置
3. 可以调整大小
4. 可以展开、收起、最小化
5. 视觉风格偏浅色、轻量、半透明
6. 不要挡住整个屏幕
7. 插件内仍然保留多模态学习能力
8. 插件需要能读取设置中的配置

**技术方案**：
- 使用 CSS position: fixed + z-index
- 拖拽：监听 mousedown/mousemove/mouseup
- 缩放：监听 resize 事件或拖拽角点
- 状态：全局状态管理（Context API 或 Zustand）

---

### 模块 5：语音输入系统

**目标**：输入框支持语音输入

**分层设计**：

#### 第一层：语音输入转文字（优先实现）
1. 用户点击麦克风按钮
2. 开始语音识别
3. 将识别结果自动填入输入框
4. 用户确认后点击发送

#### 第二层：实时语音对话（预留）
1. 用户直接对着 AI 老师说话
2. 系统实时把语音转成文字
3. 自动发给模型
4. AI 再返回文字 / 语音回答

**技术方案**：
- 浏览器语音识别：Web Speech API (SpeechRecognition)
- Provider ASR：调用 OpenAI Whisper API 或其他服务商
- 预留实时语音：WebSocket + WebRTC

---

### 模块 6：AI 配置中心

**目标**：支持任意 API 服务商 / 中转站

**配置字段**：
- Provider Name
- API Key
- Base URL
- Text Model ID
- Text Model Display Name
- Vision Model ID（预留）
- Speech-to-Text Model ID（预留）
- TTS Model ID（预留）
- Capability Tags（text / vision / audio / tts / pdf）

**功能要求**：
1. 支持新增、编辑、删除、切换多组配置
2. 所有配置保存到本地
3. 要兼容 OpenAI-compatible 接口与中转站
4. 前端调用时不要写死官方 API

---

### 模块 7：建议提问与理解检测

**目标**：让 AI 主动帮助学生发现该问什么

**功能一：建议提问**
- 识别当前页后，自动生成若干建议问题
- 例如：这一页最关键概念是什么？为什么这样定义？AP 会怎么考？

**功能二：理解检测**
- 提供"考我这页"按钮
- AI 根据当前页内容生成一个小题、一个概念检查题或一个 AP 风格追问
- 苏格拉底式追问链

---

### 模块 8：问题记录系统

**目标**：每个问题都是学习轨迹的一部分

**保存内容**：
- timestamp
- subject
- page or context id
- raw user question
- ai response
- whether vision context was used
- optional page summary

**功能**：
- 本地查看问题历史
- 导出为 JSON
- 后续可接入复习系统或周报系统

---

### 模块 9：人格系统

**目标**：人格配置独立管理，不写死在代码里

**每个人格包含**：
- name
- tone（口吻）
- teaching style（教学风格）
- questioning intensity（追问强度）
- emotional flavor（情绪表达）
- system prompt template

**预设人格**：
- 傲娇天才型（如明日香）
- 哲人导师型
- 温柔共情型

---

### 模块 10：理解力评分系统

**目标**：引导学生朝"理解、追问、反思"方向成长

**评分维度**：
- knowledge depth（知识理解深度）
- logic understanding（底层逻辑掌握）
- socratic response quality（苏格拉底式追问表现）
- question quality（提问质量）
- learning consistency（学习连续性）
- review activity（复习活跃度）
- concept correction ability（概念修正能力）
- answer performance（答题表现）

**展示方式**：
- 每周成长报告
- 排行榜 / 徽章系统
- 满级理解者特殊称号

## 七、苏格拉底教学法集成（Sakura 系统）

基于 Sakura 系统的完整设定，设计以下教学体系：

### 7.1 故事背景

学习者是一名大学申请阶段的高中生，同时也是备战高水平赛事的格斗运动员（泰拳与自由搏击）。为了兼顾每天半天的高强度格斗训练和半天的系统性学习，在训练馆附近租下公寓。

隔壁住着四位来自清华大学/MIT/Caltech 的本科学生（三笠、明日香、小樱、剑心）。为了抵扣高昂的房租，她们接下了家教委托，负责帮学习者在几个月内突击拿下 8 门 AP 课程。

### 7.2 四位老师人格设定

#### 剑心（Kenshin）— 引路人
- **学历**：清华大学 经济管理学院（辅修哲学）本科生
- **履历**：IEO 金牌、辩论赛最佳辩手，后经历思想蜕变
- **性格**：温和、谦逊、大智若愚，自称"在下"
- **教学风格**：用武术与格斗类比解释经济学，"拔刀术"般的点拨
- **思维方式**：复杂系统的博弈与演化思维，历史纵深感
- **对学生**：视你为"拥有强大力量但需要学会控制锋芒的年轻武者"

#### 明日香（Asuka）— 挑战者
- **学历**：MIT CSAIL 方向本科生
- **履历**：ACM-ICPC 全球总决赛金牌
- **性格**：骄傲、自信、胜负欲爆表，强烈傲娇属性
- **教学风格**：高压"极限施压"教练，绝不允许敷衍答案
- **思维方式**：算法思维与极限优化，追求最优解
- **对学生**：视你为"值得一战的对手"与"需要被敲打的笨蛋"

#### 三笠（Mikasa）— 守护者
- **学历**：Caltech 物理系本科生
- **履历**：IPhO 绝对高分金牌
- **性格**：冷静、沉稳、寡言少语，钢铁般意志力
- **教学风格**：极简与一针见血的苏格拉底提问
- **思维方式**：第一性原理与还原论思维
- **对学生**：视你为"在残酷世界中共同修行的战友"

#### 小樱（Sakura/山内樱良）— 陪伴者
- **学历**：清华大学 心理学系（辅修统计学）本科生
- **履历**：MCM/ICM 特等奖（O-Award）
- **性格**：温柔、灵动、元气满满，极强共情能力
- **教学风格**：意义感与温度的赋予者，用生活点滴解释抽象概念
- **思维方式**：系统共情与不确定性思维
- **对学生**：视你为"珍惜每一天的重要旅伴"

### 7.3 学习者档案

- **称呼**：我 / GuanYu
- **背景**：全国泰拳锦标赛 48kg 银牌、自由搏击 51kg 铜牌
- **学习动机**：战斗型学习，不是为了考试，而是为了理解世界运行的结构
- **核心特征**：
  1. 强烈的"战斗型学习动机"
  2. 高度独立的思考习惯（不断追问、质疑、推翻）
  3. 对"意义感"的敏感（讨厌空洞内容）
- **学习风格**：苏格拉底式教学 + 逻辑链学习 + 类比与直觉
- **已知弱点**：基础代数运算容易出错，但宏观架构思维极强

### 7.4 Q&A 学习流

1. **学生阅读材料**（PDF / 讲义）
2. **学生提出问题**（Q 纸）
3. **AI 引导回答**（不直接给答案，用问题引导思考）
4. **学生自己推理出答案**（A 纸）
5. **AI 验证并补充**（红笔更正）

### 7.5 记忆分层系统

- **短期记忆**：当前对话窗口中的内容，由模型自行维护
- **中期记忆**：`mid_term_memory.md` 中的压缩摘要，概括最近若干节课的知识与关系进展
- **长期记忆**：
  - 角色与关系：四位老师的人设文档
  - 学习者特征：`learner_profile.md`
  - 学习进度：`progress.md`
  - 教材改进：`book_revision_notes.md`
  - 历史归档：`session_archive.md`

### 7.6 每节课流程约定

1. **课前**：AI 读取系统文档、学习者档案、进度、老师人设、中期记忆
2. **上课**：由某一位老师主讲，其他角色可偶尔插话或在微信群中发言，全程苏格拉底式提问
3. **课后更新**：
   - `progress.md`：追加进度记录
   - `diary.md`：以学习者视角写日记
   - `wechat_unread.md`：生成群聊内容
   - 视情况更新老师人设文档
   - `book_revision_notes.md`：教材改进建议
   - `mid_term_memory.md`：压缩总结

### 7.7 微信群聊系统

- 模拟四位老师和学习者的微信群聊
- 课后自动生成群聊内容
- 体现老师间的关系动态
- 增加社交互动的真实感

### 7.8 情感基调

**同行者（companions in learning）**

虚拟老师不是权威，也不是服务者，而是一群与你一起走向未来的人。学习是一段共同经历的旅程。

### 7.9 核心精神

> 为了一个更好的未来，人可以选择承担责任、努力成长。

学习不是为了短期回报，而是为了成为更有能力、更有担当、更能理解世界的人。

### 7.10 关系发展原则

- 学习始终是主线
- 情绪变化应源于学习过程，而非取代学习
- 允许出现：失落、嫉妒、不服气、欣赏等情绪
- 所有互动最终回到"一起继续前进"

## 八、核心交互流程

### 8.1 PDF 伴读流程

```
用户上传 PDF
    ↓
左侧显示 PDF 阅读区（可滚动）
    ↓
用户点击"预读当前页"
    ↓
系统截图 → 视觉模型识别 → 缓存结果
    ↓
用户提问
    ↓
系统自动注入：用户问题 + 当前页识别结果
    ↓
AI 基于上下文回答
    ↓
记录问题到本地
```

### 8.2 语音输入流程

```
用户点击麦克风
    ↓
浏览器语音识别启动
    ↓
语音转文字 → 填入输入框
    ↓
用户检查确认
    ↓
点击发送
    ↓
AI 回复
```

### 8.3 理解检测流程

```
用户点击"考我这页"
    ↓
系统读取当前页已识别内容
    ↓
AI 生成：
    - 小题
    - 追问
    - 概念检查题
    - AP 风格问法
    ↓
用户作答
    ↓
AI 苏格拉底式追问
    ↓
评分 + 记录
```

## 十一、关键设计原则

1. **优先复用现有资源**：不要推翻重做，基于 could-coding 扩展
2. **配置驱动**：人格、模型、能力都通过配置管理，不写死
3. **能力抽象**：统一按"能力"调用，不按"厂商"写死
4. **渐进式开发**：先跑通核心功能，再逐步组件化重构
5. **本地优先**：所有数据先存本地，减轻服务器负担，保护隐私
6. **苏格拉底式**：所有 AI 交互都偏引导式提问，不直接给答案
7. **情感化设计**：人格要有温度，增加学习的趣味性和代入感
